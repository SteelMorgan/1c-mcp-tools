# 1c-mcp-tools

Прикладное расширение 1С для размещения MCP tools, исполняемых внутри 1С.

Репозиторий содержит только исходники расширения:

- `src/1c_ext/` - XML-выгрузка расширения 1С;
- `src/1c_ext/agents.md` - краткие правила разработки tools внутри расширения.

Собранные `.cfe`, клиентские настройки, Docker-инфраструктура и Python-прокси из этого репозитория удалены. Этот проект больше не является самостоятельным прокси-сервером или транспортным слоем.

## Место в системе

`1c-mcp-tools` - прикладной слой в общей архитектуре MCP для 1С.

Транспорт, публикация и агрегация MCP-витрины живут в других компонентах:

- `v8-runner` запускает клиент 1С и передаёт параметры подключения к менеджеру сессий;
- `web-transport-addin` даёт WebSocket-канал из клиента 1С;
- `wt-mcp-adapter` регистрирует клиентские и серверные tools в 1С и публикует их через WS;
- `v8-session-manager` (`v8sm`) агрегирует tools всех подключённых 1С-сессий и отдаёт единую MCP-витрину агенту.

Роль этого репозитория уже: хранить прикладные tools, которые будут размещаться в расширении 1С и исполняться в клиентском или серверном контексте через адаптер/менеджер.

## Что есть в расширении

Ключевые объекты:

- `mcp_Мсп_Провайдеры` - маркерная подсистема провайдера tools;
- `mcp_КонтейнерыИнструментов` - подсистема обработок, публикующих tools;
- `mcp_КонтейнерыРесурсов` - подсистема обработок, публикующих resources;
- `mcp_КонтейнерыПромптов` - подсистема обработок, публикующих prompts;
- `mcp_Метаданные` - сбор описаний tools/resources/prompts из контейнеров;
- `mcp_Выполнение` - диспетчеризация вызова в обработку-контейнер;
- `mcp_КонтейнерыПовтИсп` - получение текущего каталога tools/resources/prompts;
- `mcp_APIBackend` - legacy HTTP JSON-RPC/MCP endpoint, оставшийся от исходной реализации.

Текущие встроенные возможности:

- `list_metadata_objects` - список объектов метаданных;
- `get_metadata_structure` - структура объекта метаданных;
- `1csyntax` - ресурс со справкой по встроенному языку 1С.

## Целевая модель развития

Новые tools размещаются как обработки-контейнеры внутри расширения.

Для tool-контейнера:

1. Создать обработку с префиксом `mcp_`.
2. Включить её в подсистему `mcp_КонтейнерыИнструментов`.
3. В модуле менеджера реализовать:

```bsl
Процедура ДобавитьИнструменты(Инструменты) Экспорт
	// Описать tools и их JSON schema через mcp_Метаданные.
КонецПроцедуры

Функция ВыполнитьИнструмент(ИмяИнструмента, Аргументы) Экспорт
	// Выполнить нужный tool и вернуть результат.
КонецФункции
```

Для resources и prompts используются те же принципы:

- контейнер resources включается в `mcp_КонтейнерыРесурсов`;
- контейнер prompts включается в `mcp_КонтейнерыПромптов`;
- описание и исполнение идут через экспортные методы контейнера.

Подробности текущего внутреннего API описаны в [`src/1c_ext/agents.md`](./src/1c_ext/agents.md).

## Происхождение

Технически этот репозиторий начался как форк проекта [RooLee10/1c-mcp-tools](https://github.com/RooLee10/1c-mcp-tools), который, в свою очередь, основан на оригинальном проекте [vladimir-kharin/1c_mcp](https://github.com/vladimir-kharin/1c_mcp).

Авторам исходных проектов - уважение и благодарность за основу и идею.

При этом в текущем репозитории существенная часть функциональности уже отломана от исходной реализации, переработана или будет заменяться. Поддерживать зависимость от upstream-проектов и сохранять совместимость с ними не планируется.

Дальше проект развивается как самостоятельное прикладное расширение в текущей архитектуре: площадка для размещения 1С MCP tools клиентского и серверного исполнения.

## Что не входит в проект

В этом репозитории больше нет:

- Python-прокси;
- Docker-образа прокси;
- `docker-compose.yml`;
- готового `build/*.cfe`;
- примеров настроек MCP-клиентов;
- batch-скриптов запуска HTTP-сервера.

Эти функции относятся к инфраструктуре запуска, транспорта и агрегации. Здесь остаётся только 1С-расширение с прикладными tools.
